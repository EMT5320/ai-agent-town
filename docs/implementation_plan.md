# 实施计划

## 阶段 0：方向冻结与技术 Spike

目标：验证 Godot + Python Agent Server 的协作模式。

### 交付物

- Godot 空项目或最小客户端。
- 一张测试小镇地图。
- 一个可移动玩家角色。
- 3 个 NPC 可视化对象。
- Godot 从后端读取 NPC 状态。
- 玩家点击 NPC 后可以发起对话。
- 后端返回对话结果并写入事件流。
- Debug 控制台能看到本次交互日志。

### 验收标准

- Godot 能正常运行并连接 Python 后端。
- 玩家与 NPC 对话链路完整。
- 后端保留 Agent 决策与记忆写入记录。
- 项目结构证明后续可以扩展。

## 阶段 1：Agent Valley：第一天

目标：完成可录屏、可讲解、可玩的垂直切片。

### 内容范围

- 1 个玩家角色。
- 6 个 NPC。
- 3 个地点：农场、广场、酒馆。
- 1 个完整游戏日。
- 1 个小镇事件。
- 玩家可移动、聊天、送礼。
- NPC 有日程、状态、关系和记忆。
- 夜晚生成 NPC 日记摘要。

### 关键系统

#### 玩家系统

- 移动
- 交互
- 对话
- 送礼
- 查看 NPC 简介

#### NPC 系统

- 日程
- 地点移动
- 基础工作行为
- 玩家交互响应
- NPC 间低频互动
- 记忆写入
- 夜间反思

#### 事件系统

- 小镇事件触发
- NPC 主观记忆生成
- 关系变化
- 事件日志
- 事件 CG 展示

#### Debug 系统

- Agent 决策记录
- Prompt / messages
- 原始输出
- 解析行动
- 工具执行结果
- 记忆变化
- 关系变化

### 美术资产

使用 Codex 生图能力生成：

- 小镇概念图 1 张
- 农场背景 1 张
- 广场背景 1 张
- 酒馆背景 1 张
- 玩家角色概念图 1 张
- 6 个 NPC 头像或半身像
- 1 张事件 CG
- UI 面板风格图 1 组

## 阶段 2：可解释 Agent 游戏化

目标：让 Agent 系统具备更强的可解释展示能力。

### 重点功能

- NPC 记忆面板
- NPC 对玩家的印象变化
- NPC 对其他 NPC 的关系摘要
- 事件影响链路
- 每日小镇总结
- Debug 回放时间线
- 实验数据导出

### 展示重点

- 同一事件被不同 NPC 不同地理解。
- 玩家行为能影响后续对话。
- NPC 的历史记忆会影响下一次决策。
- Debug 面板能解释 NPC 为什么这样行动。

## 近期任务建议

### 第 1 批任务

1. 确认 Godot 作为主客户端方向。
2. 安装并验证 Godot 运行环境。
3. 在仓库中创建 `clients/godot/`。
4. 搭建最小 Godot 场景。
5. 给后端补一个适合 Godot 拉取的 `GET /api/world/state` 数据格式。
6. 做玩家与 NPC 对话的最小闭环。

### 第 2 批任务

1. 设计 6 个 NPC。
2. 设计 3 个地点。
3. 制作第一批 Codex 生图提示词。
4. 生成并筛选视觉资产。
5. 建立资产命名与清单。
6. 在 Godot 中显示角色和地点。

### 第 3 批任务

1. 增加 NPC 日程。
2. 增加送礼动作。
3. 增加记忆写入规则。
4. 增加夜间反思。
5. 增加事件 CG 展示。
6. 增加 Debug 时间线。

## 风险控制

### 规模风险

首版限制在 6 个 NPC、3 个地点、1 个游戏日。每个扩展功能都必须服务于垂直切片。

### 成本风险

LLM 调用分层：

- 普通移动和日程：规则。
- 玩家关键对话：LLM。
- 重大事件反应：LLM。
- 夜间反思：LLM。

### 引擎风险

先做 Godot Spike。若 Godot 工作流明显阻塞，再评估 Phaser 或纯 Web 方案。

### 美术风险

先生成少量核心资产，确定风格后再批量生成。资产进入项目必须有用途、命名和来源记录。
