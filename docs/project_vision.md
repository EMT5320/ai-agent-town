# 项目整体愿景：Agent Valley

> 本文是项目后续推进的核心依据。后续设计、实现、取舍和阶段验收都优先对齐这里定义的方向。

## 一句话定位

`Agent Valley` 是一个精美可玩的生活模拟 RPG 原型：玩家进入一座由 LLM Agent 居民自主运转的小镇，与拥有独立人设、日程、目标、关系和记忆的 NPC 共同生活。

## 项目目标

把当前 `AI Agent 小镇实验室` 从“多 Agent 观察台”升级为“可游玩的多 Agent 社会模拟游戏”。

最终展示形态应同时满足三类目标：

1. **像游戏**：玩家能移动、探索、聊天、送礼、触发事件，并感受到小镇居民在生活。
2. **有技术深度**：NPC 行为由 Agent 系统驱动，行动可追踪，记忆可解释，事件可回放。
3. **适合作品集与论文**：画面精美，架构清晰，有可展示 Demo，有可导出的实验数据。

## 核心体验

玩家进入小镇后，可以：

- 操控自己的角色在地图中移动。
- 与 NPC 对话，影响对方对玩家的印象。
- 送礼、参加活动、触发特殊事件。
- 观察 NPC 根据日程、目标、情绪和记忆自主行动。
- 看到小镇中的事件被不同 NPC 以不同方式记住。
- 在 Debug / 研究控制台中查看 Agent 输入、模型输出、工具调用、记忆写入和世界状态变化。

## 核心创新点

### 1. LLM 驱动的 NPC 社会

每个 NPC 都是一个独立 Agent，具备：

- 人格设定
- 长期目标
- 今日计划
- 当前位置
- 情绪与体力
- 职业与经济状态
- 与玩家及其他 NPC 的关系
- 短期记忆
- 长期记忆
- 对事件的主观解释

### 2. 行动即工具调用

游戏内行为统一表达为可追踪行动：

```text
moveTo(location)
talkTo(target, topic, message)
work(job)
buy(item)
sell(item)
giveGift(target, item)
attendEvent(event)
remember(event)
reflect()
```

玩家看到的是 NPC 在生活；系统内部记录的是 Agent 决策、行动执行、状态变化和记忆更新。

### 3. 主观记忆管理

同一个事件会进入不同 NPC 的不同记忆上下文：

- 在场者获得直接记忆。
- 相关者获得高权重记忆。
- 旁观者获得低权重记忆。
- 亲密关系会放大情绪影响。
- 敌对关系会改变事件解释。
- 夜间反思会把短期记忆沉淀为长期摘要。

示例：

```text
玩家送给医生一束花。

医生：记住玩家体贴，关系上升。
暗恋医生的居民：记住玩家和医生关系变亲近，产生嫉妒。
酒馆老板：把它当作小镇八卦，在后续对话中传播。
```

### 4. AI 辅助美术资产管线

Codex 应用内生图能力作为开发期资产生产工具，用于生成：

- NPC 头像与立绘
- 场景背景
- 地图概念图
- 事件 CG
- UI 面板
- 图标与物品插画
- 宣传图与论文配图

游戏运行时使用静态资源，不依赖图片生成 API。

## 产品边界

### 第一阶段聚焦内容

首个可玩版本聚焦“第一天”垂直切片：

- 1 个玩家角色
- 6 个 NPC
- 3 个地点：农场、广场、酒馆
- 1 个完整游戏日
- 1 个小镇事件
- 玩家可移动、聊天、送礼
- NPC 有日程、对话、记忆和关系变化
- 夜晚生成 NPC 日记或反思摘要

### 暂缓内容

以下内容等垂直切片稳定后再扩展：

- 大规模地图
- 四季系统
- 完整农场经营
- 复杂经济系统
- 恋爱结婚系统
- 多年人口演化
- 战斗系统
- 联机玩法
- 运行时生图 API

## 技术路线共识

### 客户端

主游戏客户端优先评估 Godot：

- 负责地图、玩家控制、NPC 表现、动画、UI 和交互。
- 优先使用 GDScript，降低 Web 导出和跨平台风险。
- 第一阶段以 Windows 桌面 Demo 为主要目标。

### 后端

保留并演进现有 Python Agent 后端：

- 世界状态
- 时间推进
- NPC 调度
- LLM Provider
- 行动解析与执行
- 记忆系统
- 关系系统
- 事件流
- Debug 数据
- 实验导出

### Debug / 研究控制台

现有网页前端后续转为：

- Prompt 查看器
- 模型输出查看器
- 工具调用时间线
- 记忆写入检查器
- 关系图谱观察器
- 事件注入工具
- 实验数据导出入口

## 成功标准

### 体验成功

- 玩家能在 3 分钟内理解：这是一个由 AI 居民自主生活的小镇。
- NPC 对玩家行为有持续反应。
- 同一事件能引发多个 NPC 的不同记忆和后续行为。
- Demo 截图具备作品集展示价值。

### 技术成功

- 每次 NPC 决策都有可追踪链路：上下文、模型输出、行动、执行结果、记忆更新。
- Agent 后端与游戏客户端职责清晰。
- 运行时可以在规则 Provider 与云端 Provider 间切换。
- 实验数据可以导出，支持论文分析。

### 工程成功

- 第一版垂直切片可以稳定运行。
- 新增 NPC、地点、事件和工具行动有明确扩展路径。
- 美术资产有统一命名、来源记录和导入流程。
- Debug 信息不会污染玩家主游戏体验。

## 推进原则

1. **先做可玩垂直切片**：优先证明“玩家 + NPC Agent + 记忆 + 事件”的闭环。
2. **表现层与 Agent 层分离**：游戏客户端负责体验，Python 后端负责智能体系统。
3. **开发期用生图，运行时用静态资产**：降低成本和工程复杂度。
4. **每个行为都可解释**：Agent 行动必须能回溯到上下文、输出和工具调用。
5. **先少量 NPC 做深**：首版 6 个 NPC 比 30 个浅层 NPC 更适合展示。
6. **作品集观感优先**：每个阶段都要产出可截图、可录屏、可讲解的成果。
